# クライアント対応ドキュメント一覧

LINE入退室通知システムのクライアント対応に必要なドキュメントをまとめています。

---

## 📚 ドキュメント構成

### 1. 📋 **情報チェックリスト** - `client-information-checklist.md`
**用途**: クライアントから取得すべき情報の完全リスト

**内容:**
- LINE公式アカウント情報
- 教室・運営情報
- 生徒・親御さん情報
- NFCカード情報
- 通知内容のカスタマイズ要件
- 運用ルール
- デプロイ・インフラ情報
- テスト・運用開始計画

**対象**: 開発者・プロジェクトマネージャー  
**ページ数**: 詳細版（チェックボックス付き）

---

### 2. 📧 **依頼テンプレート** - `client-request-template.md`
**用途**: クライアントに実際に送信する依頼書

**内容:**
- 最優先で必要な情報（5項目）
- 後日でも構わない情報
- スケジュール感
- 返信フォーマット

**対象**: クライアント様に直接送信  
**ページ数**: 簡潔版（A4 2ページ程度）

---

### 3. ❓ **FAQ** - `client-faq.md`
**用途**: クライアントからよくある質問への回答集

**内容:**
- LINE公式アカウントについて（Q1〜Q3）
- 保護者の登録について（Q4〜Q7）
- NFCカードについて（Q8〜Q11）
- 通知について（Q12〜Q14）
- セキュリティ・プライバシー（Q15〜Q17）
- 費用について（Q18〜Q19）
- 運用について（Q20〜Q23）
- カスタマイズ・拡張（Q24〜Q26）
- トラブル対応（Q27〜Q29）
- その他（Q30）

**対象**: クライアント様・営業担当  
**ページ数**: 全30問

---

## 🎯 使い方ガイド

### シーン別の使い分け

#### 📞 **初回ヒアリング時**
1. `client-request-template.md` を送付
2. クライアントから返信を受領
3. `client-information-checklist.md` で情報の漏れをチェック

#### 💬 **質問対応時**
- `client-faq.md` を参照して回答
- 新しい質問があれば追加

#### 🔄 **プロジェクト進行中**
- `client-information-checklist.md` で進捗管理
- 未取得の情報を定期的に確認

---

## 📊 情報収集の優先順位

### 🔴 最優先（これがないと開発不可）
1. ✅ LINE公式アカウント（または作成の許可）
2. ✅ 生徒リスト（最低限: 名前・学年）
3. ✅ 親御さんの連絡先（最低限: 氏名・電話番号）
4. ✅ 通知内容の基本方針

**→ これらが揃えば開発開始可能**

---

### 🟡 重要（早めに確認したい）
5. ⚠️ NFCカードの準備状況
6. ⚠️ 入退室の運用ルール
7. ⚠️ セキュリティ・プライバシー方針

**→ 1週間以内に確認推奨**

---

### 🟢 追加要件（後から調整可能）
8. 📝 通知メッセージのカスタマイズ
9. 📝 ポイントシステムの詳細
10. 📝 専用ドメイン・デザイン

**→ 運用しながら調整可能**

---

## 📅 標準的なスケジュール

```
Day 1-2: 情報収集
  ├─ client-request-template.md を送付
  ├─ クライアントから返信
  └─ client-information-checklist.md で確認

Day 3: LINE設定
  ├─ LINE Developers設定
  ├─ チャネルアクセストークン取得
  └─ Webhook設定

Day 4-5: データ投入
  ├─ 生徒データをデータベースに登録
  ├─ 親御さんデータを登録
  └─ NFCカードとの紐付け

Day 6-8: テスト運用
  ├─ テストユーザーで動作確認
  ├─ 親御さんへの通知テスト
  └─ クライアントと一緒に検証

Day 9: 本番運用開始
  ├─ 親御さんへの案内配布
  ├─ LINE友だち追加の促進
  └─ 運用開始 🎉
```

**最短: 1週間〜2週間で運用開始可能**

## 💡 Tips

### クライアントとのコミュニケーションのコツ

#### 1. **段階的に情報収集**
❌ 一度に全部の情報を要求
✅ 最優先情報 → 重要情報 → 追加要件の順

#### 2. **具体例を示す**
❌ 「通知内容を教えてください」
✅ 「以下のような通知でよろしいでしょうか？」
```
太郎さんが入室しました。
時刻: 2026/01/06 15:30
```

#### 3. **選択式の質問を活用**
❌ 「どうしますか？」
✅ 「以下のどちらがよろしいでしょうか？
   □ 入室時のみ通知
   □ 入退室両方通知」

#### 4. **実際のデータでテスト**
- 開発初期にテスト用の実データ（1〜2件）を取得
- 実際の運用イメージを共有

#### 5. **柔軟に対応**
- 後からカスタマイズ可能な設計
- 運用しながら改善していく姿勢
